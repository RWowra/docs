Runtime Statistics Configuration

The runtime statistics configuration helps to configure the settings for analyzing the performance statistics of a simulation during run time in ETAS EE.

When a new system is created in COSYM, the default runTimeConfig.json file is created at <COSYM project location>\<project name>\Systems\<system name>.

The content of the default runTimeConfig.json file is shown below:

Copy 
{
    "runtimeStatistics": {
        "enable": false,
        "baseTime": "WALL_CLOCK_TIME",
        "logFileName": "C:/ETAS/LogFiles/cosym/SilStatisticsLogging",
        "logFormat": "TXT",
        "logInterval": -1.0
    }
}

 

Field Default value Accepted Values Description
enable false true Enables performance statistics measurement
false Disables performance statistics measurement
baseTime WALL_CLOCK_TIME WALL_CLOCK_TIME Computes time in wall clock
CPU_TIME Measures the total time elapsed in CPU time
logFileName C:/ETAS/LogFiles/cosym/
SilStatisticsLogging 		<Specify the location to generate the log file> Path followed by file name
logFormat TXT TXT Logs in plain text file format
JSON Logs in json file format
logInterval -1.0 0 or negative value (<=0) Logs the data only when the simulation is stopped
positive value (>0) Overwrites the log file during the simulation at the given interval

The runtime statistics data is generated during simulation when you make the settings as "enable": true, in the generated runTimeConfig.json file.  By default, it is false. Make sure that the changes are made in the *.json file before the system build.

It is also possible to make the same settings through GUI of ETAS Experiement Environment application after the build process. Refer to "Enabling Runtime Statistics" section in "ETAS_Experiment_Environment.pdf" document. The settings made in the ETAS EE retain until the next code generation and build process only.

When it is enabled, the log file is generated in the specified path and format which contains the data based on the inputs provided in the *.json file. You can use this file for further analysis.

If the same function is mapped under different tasks of a model instance, the performance statistics log will have only one entry of the function with higher priority.
