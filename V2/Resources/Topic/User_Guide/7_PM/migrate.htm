Migrating a Project

COSYM supports migrating the projects created in the older version of COSYM into the latest version. While migrating a project, make sure that the project folder name is same as the project file name (*.csp).

It migrates the projects created from COSYM older versions. Refer to Supporting Versions of COSYM for Migration. While migrating, all the COSYM artifacts and the file information will be updated to the COSYM current version. Once the project and its artifacts are migrated, the changes will not be reverted. COSYM does not support for backward compatibility.

To migrate a project to a newer version

  1. Open a COSYM V3.2.0/V3.1.0 project. Refer to Migrating a Project.
  2. The Windows File Explorer is displayed to select a project.
  3. Browse and select the project and click OK.
  4. The "Migrate Project" dialog box is displayed.
    The selected project name and path are displayed in the "Project Name" and "Current Path" fields respectively.

Fig. 90: Migrate Project dialog box

  • Select the destination path in the "Destination Path" field to migrate a project.
  • Click Migrate.
  • The "Replace or skip Project" dialog box is displayed if the selected project is already available in the destination.
  • Click either "Replace the project in the destination" to replace the existing project or "Skip this project" to ignore the project.
  • The selected project is migrated along with its artifacts. The files are updated to the current version.

Note 

If the *.slxip files are used in a project created in the earlier version, might not work with the current version. In such a case, you need to regenerate the code in MATLABÂ® again and re-import in the migrated COSYM project.

Note 

If you are facing an issue while migrating a project, refer to Unable to Migrate a Project.

Note 

If any missing parameters in an FMU model after migration, refer to Parameter Files Not Created for FMU After Migration.

Handling of unmapped processes during build of the migrated project

During migration, if the "Suppress build warning" was enabled, then "[1]: ignore" option is selected. Refer to page [1]: ignoreIf this option is selected, all unmapped processes are ignored and the build process starts without displaying the dialog box. But, a warning message is displayed to inform you that there are unmapped processes which are still available in a system and those processes will not be executed during the build and this could lead to a different simulation result. for more details.

If "Suppress build warning" was disabled, then "[2]: Ask everytime" option is selected. Refer to [2]: Ask everytimeBy default this option is selected. When you start the build process for a system, a dialog box is displayed if any unmapped processes are available in that particular system. The dialog box prompts you either to map and build (same action as [0]:Automap) or to build without mapping (same action as [1]:ignore) the unmapped processes. for more details.
